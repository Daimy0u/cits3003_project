    Project

    The main project task

    You are required to complete an implementation of a simple scene editor that allows a collection of objects to be arranged in a scene and various properties of them to be changed, such as colour, shininess and texture. Below is a list of the specific features (tasks) that must be implemented in the scene editor, along with sample videos to illustrate their functionality.
    • Download the starting point file cits3003_project.zip and follow the instructions in cits3003_project_readme to build the project files.
    • We've also included detailed documentation of the project in the file cits3003_project_documentation. While it's not mandatory to read the entire documentation, doing so will significantly enhance your understanding of the project's overall structure and the details of various functions.

    Specific tasks that your scene editor should implement

    Your scene editor needs to implement all the functionalities described in the items A to I below.

    Don't panic if the videos provided alongside the items seem beyond what you have done in the labs. The project tasks have been chosen to cover all the relevant concepts without requiring you to implement too much extra. Thus, much of the functionalities are already implemented in the skeleton code, and some of the tasks require similar code to what is in the skeleton code.

    Also, you are allowed to improve upon the specification and implement some items differently, but you should explain in your report why it is an improvement.

    1. Panning camera

      The skeleton code draws the ground, and a sphere showing the position of a single point light source; however, it has the default panning camera always facing the same way. You can move it forwards and backwards using the scroll wheel, via the PanningCamera's distance variable, but there's no way to move it otherwise. You should fix this problem in the code. The variables pitch and yaw are already set appropriately when moving the mouse with the right button down. The focus_point variable is set appropriately when dragging with middle mouse. Modify the PanningCamera's update function so that the camera rotates around the focus_point according to these variables, in the same way as the sample solution shown in the video .

      The icon in the bottom left show what mouse button(s) is/are currently down.

      Do not panic when your cursor disappears while dragging it, that is intentional to allow you to drag infinitely without hitting the edge of your monitor. That has been disabled for this video , so you can see the direction I am moving the cursor.

      watch video
      Your PanningCamera should do the following:
      • The camera should start with the same orientation as at the start of the video. The camera can also be reset to this orientation by pressing the R key.
      • When the right mouse button is held down and dragged horizontally across the window, it should rotate the scene about the axis that is vertical to the ground plane.
      • Dragging the right mouse button vertically up (or down) across the window should change the elevation angle of the camera looking at the ground plane. Correspondingly, this tilts the entire scene up (or down).
      • Dragging the middle mouse button horizontally or vertical should pan the camera.
      • The mouse scroll should move the camera in and out of the focus_point

    2. Entity rotation

      Similarly, changing the location and scale of scene elements is already implemented, via the position and scale fields of the LocalTransformComponent class. However, the euler_rotation field in this class does not affect what's drawn, even though it is appropriately set to contain the rotations around the X, Y and Z axes, in radians. Modify the components calc_model_matrix method so that scene elements using this component (EntityElement, EmissiveEntityElement & AnimatedEntityElement) are appropriately rotated when drawn, like as shown in the sample solution video.

      watch video
      As shown in the video, your scene editor should do the following: With the camera yaw set to 0, so that the world axes align to the standard directions of OpenGL NDC (Normalized Device Coordinates).
      • Increasing the x angle should rotate the object about the x-axis (parallel to the ground plane and pointing to the right)
      • Increasing the y angle should rotate the object about the y-axis (vertical to the ground plane and pointing up).
      • Increasing the z angle should rotate the object about the z-axis (parallel to the ground plane and pointing to the out of the screen).
      • The order of composition of these angles will give different results, so just do something sensible.

    3. Clipping plane

      In the skeleton code, triangles are "clipped" (not displayed) if they are even slightly close to the camera. Fix the update function of the PanningCamera and FlyingCamera so that the cameras can give more "close up" views of objects.
      The relevant properties are already exposed to the UI, just not being used.

    4. Material properties

      Implement some new UI controls for adjusting the ambient, diffuse, emissive and specular tints (where appropriate), as well as the amount of shininess of EntityElement, EmissiveEntityElement and AnimatedEntityElement. Recall that the shine value (shininess) can be adjusted up to at least 100 to support highly reflective materials. Modify the material UI functions. The code to use these (new UI controls) to affect the shading via the calculation for the light is already implemented in the skeleton code provided.

      Follow the corresponding implementations of other similar UI controls (hint: Look at PointLightElement), and pay attention to how the alpha channel is used, as it's not actually used as alpha.

    5. Texture scaling

      Implement the ability to scale the textures used by EntityElement and AnimatedEntityElement, and add the ability to change these scales in the UI. (You can optionally also add this for EmissiveEntityElement)
      This can be a single scale for both diffuse and specular or separate scales, and can be a single uniform scale or per axis.
      Either way, care must be taken to also make sure the new field(s) is(/are) saved and loaded correctly, you can just mirror how this is done for the other fields.

      Hint: Look at how the other material properties (e.g., shininess) are implemented. You won't need to repeat much code between the two scene element types. Just like how there was no need to repeat code for part B. Also I'd suggest using default values for the scale so that you don't need to update all the structs.

    6. Light attenuation

      Modify the point light calculation in res/shaders/common/lights.glsl so that the light reduces with distance - exactly how it reduces is up to you, but you should aim for the reduction to be noticeable without quickly becoming very dark, similar to the sample solution.

      Remember you can use "Reload Shader Files" under the "Shader Options" menu to test your shader changes without needing to restart the program.

      CHALLENGE: If you want a challenge, then try to parameterize the "attenuation" of the point light, and upload those parameter(s) to the shader via uniforms. Add a UI control to edit these parameter(s). Make sure that these parameter(s) are correctly saved and loaded when saving and loading the scene. Hint: Look at how the lightâ€™s color is currently handled as this will give you a blueprint for how to handle this task. Attempting this challenge can result in a bonus(+1) mark.

    7. Shading

      Move the light calculations from the vertex shader to the fragment shader, so that the directions are calculated for individual fragments rather than for the vertices. This needs to be done for both entity and animated_entity (but NOT emissive_entity since it isn't lit). This should have a very noticeable effect on the way the light interacts with the ground (especially when the ground has a plain texture map), since the ground only has vertices at the corners of a coarse grid. Also don't forget to change the set_vert_define calls with set_frag_define calls, otherwise NUM_PL would not be defined in the right shaders.

      watch video

    8. Light source

      Add a second light type to both the C++ code and the shaders. The second light should be directional, i.e., the direction that the light hits each surface is constant and independent of the position of the surface. In particular, using direction vector in the shader to describe the light direction, which will be easier than trying to use angles like pitch and yaw in the shader (though it is possible).

      The lights' colour, direction, etc., should be editable from the UI, and a similar approach should be used as for point lights to support multiple directional lights.

      Also, it would be beneficial to have an in-world entity (like how point light adds a sphere) that visually indicates the direction of the directional light source. The specific implementation of this is open to interpretation and creativity.

      Also make sure that this light is saved and loaded correctly when saving and loading the scene.

      This is somewhat of a big task, so look at Lights.h, Lights.cpp, MasterRenderScene, PointLightElement, lights.glsl, the fragment shaders of both entity and animated_entity, both EntityRenderer and AnimatedEntityRenderer and EditorScene and probably some other places to see how the point light is handled and basically repeat all that for the directional light.
